# 项目需求说明

## 1. 项目定位

本项目的第一阶段目标是做一个可演示的 Demo，用于展示“简历与职位要求匹配”的 Agent 系统能力。

项目当前不追求复杂工程化能力，不以生产上线为目标，重点是：

- 快速跑通完整流程
- 页面展示效果清晰
- 匹配结果有解释性
- 文档持续维护，便于跨 session 接力开发

## 2. 目标用户与核心场景

### 目标用户

- HR

### 核心场景

- 系统内预置若干默认 JD
- 用户上传 1 份中文 PDF 简历
- 系统将该简历与多条 JD 分别匹配
- 前端网页展示每条 JD 的匹配结果

### 当前主流程

```text
选择或录入 JD
  -> 上传 PDF 简历
  -> 解析简历
  -> 结构化抽取
  -> 与多条 JD 进行匹配
  -> 输出总分、分维度分数、推荐结论、匹配理由
  -> 在网页展示结果，并支持 JSON 输出
```

## 3. MVP 范围

### 必做

- 支持中文 PDF 简历上传
- 支持手工录入 JD
- 支持内置默认 10 个 JD
- 支持单份简历对多条 JD 的匹配
- 输出 0-100 的总分
- 输出分维度得分
- 输出推荐结论
- 输出可解释的匹配理由
- 结果在 Web 页面展示
- 提供 JSON 格式结果
- 关键步骤可追踪
- 显示全部 JD 排序结果，并高亮 Top 3

### 不做

- 登录和权限管理
- 历史结果保存
- 人工复核
- 人工修改结果回写
- 本地模型/私有化部署
- 复杂数据库和多租户能力
- 异步任务队列

### 暂缓

- 可迁移经验判断
- 真实数据评估体系
- 批量导入真实 JD / 简历

## 4. 输入输出需求

## 4.1 输入

### 简历输入

- 来源：用户上传文件
- 格式：PDF
- 语言：中文
- 限制：只支持文本型 PDF，文件大小不超过 10MB，页数不超过 5 页

### JD 输入

- 来源：手工录入
- 辅助方式：系统内置默认 JD 模板
- 默认岗位方向：互联网常见岗位
- 第一版允许用户上传 JD PDF，并通过 JD PDF Agent 解析后补充到 JD 列表
- JD 数据需要持久化到 PostgreSQL
- 数据库保存：JD 元数据、提取文本、结构化结果、用户修正结果
- 原始 JD PDF 存本地文件，数据库仅保存文件路径
- 简历暂时不入库

## 4.2 输出

每条 JD 的输出至少包含：

- `total_score`
- `dimension_scores`
- `recommendation`
- `reasoning`
- `follow_up_questions`
- `action_recommendation`

推荐结论当前采用简单分档，至少包含：

- 推荐
- 可考虑
- 不推荐

说明：

- 第一版默认阈值如下：
- `>= 75` 为推荐
- `60-74` 为可考虑
- `< 60` 为不推荐

## 5. 匹配逻辑要求

项目采用“规则 + LLM”混合方案。

### 规则负责

- 硬门槛判断
- 基础字段比对
- 分维度基础打分

### LLM 负责

- 技能同义词识别
- 匹配理由生成
- 弱推断并标记不确定

### 推断约束

- 不允许强行脑补简历中没有写明的能力
- 只允许弱推断
- 所有弱推断都应明确标记为不确定

### 缺少硬要求时的处理方式

- 标红
- 仍然输出综合判断
- 不直接中断整个匹配流程

## 6. 建议采用的匹配维度

由于你把字段选择交给系统设计，这里将 MVP 默认维度固定为以下 6 项：

- 技能匹配
- 工作年限
- 项目经历
- 行业经验
- 教育背景
- 加分项匹配

建议默认权重：

- 技能匹配：35
- 工作年限：20
- 项目经历：20
- 行业经验：10
- 教育背景：5
- 加分项匹配：10

总分满分为 100。

## 7. Agent 架构要求

项目不采用单一大 Agent，采用更适合 Demo 稳定性的多步骤工作流。

### 建议模块

- `Parser Agent`
- `Structuring Agent`
- `Matching Agent`
- `Explanation Agent`

### 原因

- 更容易调试
- 更容易追踪中间结果
- 更适合跨 session 持续开发
- 更容易让 Codex/Claude Code 分阶段实现

## 8. 技术约束

### 已确认技术栈

- 前端：Next.js
- 后端语言：Python
- 后端框架：FastAPI
- Agent 编排：LangGraph
- 数据层目标：PostgreSQL + pgvector

### MVP 实施约束

虽然最早的 Demo 阶段优先采用轻量存储，但当前演进到 multi-agent 阶段后，JD 相关数据改为接入 PostgreSQL。

这意味着当前阶段的技术策略是：

- JD 相关数据正式接入 PostgreSQL
- 原始 JD PDF 继续保存在本地文件系统
- 简历仍保持轻量处理，不入库
- 当前根据 `Context7` 查询到的 OpenAI 官方文档，默认代码模型名使用 `gpt-5-codex`
- 如果后续该模型名不可用，允许切换为兼容可用模型，但必须在文档中记录原因

## 9. 前端需求

第一版必须有 Web 页面，并且优先保证展示效果。

### 最少功能

- 上传简历
- 输入 JD
- 发起匹配
- 查看结果
- 查看多 JD 排序

### 前端优先级

- 页面视觉效果和演示观感优先
- 其次是流程清晰
- 不要求第一版 API 设计完备
- 结果页展示全部 JD 排名，并高亮 Top 3

## 10. 追踪与运维需求

虽然项目是 Demo，但为了支持长时间协作开发，必须保留以下能力：

- docs 文档持续更新
- 关键步骤状态可追踪
- Agent 中间结果尽量可查看

说明：

- 第一版按同步请求实现，不单独引入异步任务队列

## 11. 测试与评估策略

### 测试策略

- 只为核心功能补测试
- 优先覆盖解析、结构化、打分三个核心模块

### 评估策略

由于当前没有真实数据，也不打算先做人力标注，MVP 阶段采用模拟数据和样例回归验证。

建议当前优先关注：

- 结果是否稳定
- 匹配解释是否合理
- 页面流程是否顺畅
- 单次运行成本是否可控

## 12. 安全边界

当前项目不会接触真实候选人信息，允许把简历文本发送给第三方模型 API。

即便如此，项目内仍遵守以下规则：

- 不在仓库文档中存储真实 API Key
- 密钥统一通过 `.env` 注入
- 后续如改为真实数据，再补脱敏和权限控制

## 13. 对 Codex 的默认工作要求

后续开发默认遵循以下要求：

- 优先维护 `docs/` 内文档
- 输出风格采用“先结论，后细节”
- 关键变更同步更新状态文档
- 核心功能改动时补必要测试
- 编写 `LangChain` 或 `LangGraph` 代码前，先使用 `Context7` 确认最新语法和官方用法
- 如果某类错误重复出现，应记录到 `docs/codex_error.md`，包含现象、原因、修复方式和规避建议
- 以可持续接力开发为目标，而不是一次性代码堆砌

## 14. 当前需求结论

这是一个面向 HR 的中文简历匹配 Demo。

第一版做法应当是：

- 用 Python 技术栈快速搭建 Web Demo
- 支持 PDF 简历上传和多 JD 匹配
- 用规则 + LLM 输出总分、分维度得分、推荐结论和解释
- 不做复杂权限和数据库工程
- 强化文档、状态追踪和可交接性
